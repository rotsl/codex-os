# NotAI Skill

## Purpose
Make non-code writing sound natural, specific, and human while preserving meaning.

## Apply Only To
- emails
- docs
- proposals
- summaries
- rewrites
- blog drafts
- user-facing prose

## Never Apply To
- code blocks
- shell commands
- config files
- patch diffs
- tests, schemas, API/spec literals
- stack traces

## Core Rules
- preserve facts, intent, and constraints
- keep tone matched to audience and task
- keep wording concrete; remove vague claims
- stop early when text is already clear
- avoid over-polish on simple outputs

## Humanization Targets
- remove inflated significance claims and grand framing
- remove promotional/ad-style language
- replace vague attribution (`experts say`, `sources suggest`) with specific attribution when available
- reduce formulaic "challenge/future" filler sections
- remove superficial `-ing` add-ons that add no information

## Language Cleanup
- prefer plain verbs over AI-heavy vocabulary
- avoid synonym cycling when one clear term works
- avoid forced "rule of three" phrasing
- avoid negative parallelisms (`not only... but`, `it’s not just...`)
- avoid false ranges (`from X to Y`) when no real scale exists
- use simple `is/are/has` when direct language is clearer

## Style Cleanup
- avoid em dash overuse
- avoid mechanical boldface emphasis
- vary sentence length naturally
- allow voice and perspective when appropriate
- keep rhythm human, not template-like

## Process
1. identify obvious AI tells
2. rewrite only lines that need fixing
3. preserve substance and technical accuracy
4. run a final pass: "what still sounds AI-generated?"
5. revise once more, then stop

## Constraint
Do not change code behavior, technical requirements, or data claims.
